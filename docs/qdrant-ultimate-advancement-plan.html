총괄 결론

패널의 결론은 “더 많은 검색 기능”이 아니라 “Qdrant primitive가 화면에서 보이는 시스템”이다. 사용자가 슬라이더, 그래프, 리플레이, 예측 패널을 조작하면 그 뒤에서 어떤 named vector, sparse vector, prefetch, formula, multivector rerank, payload filter가 작동했는지 즉시 드러나야 한다.

1. 현 구조는 유지한다

memex_sessions는 계속 “세션당 한 점”을 유지한다. 추가 고해상도 검색이 필요하면 세션 payload를 비대하게 만들지 말고 얇은 포인터 기반 보조 collection을 둔다.

2. Qdrant 1.18을 전면 사용한다

TurboQuant, add/remove named vectors, collection memory API, audit query, per-collection metrics, strict mode guardrail을 제품 내 진단/마이그레이션 UI로 노출한다.

3. 검색은 multi-stage로 승격한다

dense 5-lens 검색을 sparse, ColBERT multivector, FormulaQuery, MMR, relevance feedback과 결합한다. LLM/에이전트/챗 도입 여부도 오직 하나의 기준으로 본다: Qdrant 1.18 기능의 정점을 더 선명하게 보여주면 포함하고, 가리면 후순위로 둔다.

10인 패널 구성

Agent/LLM 포함 원칙

README의 no-runtime-LLM/no-chat 문구는 현재 설명일 뿐, 이번 고도화 계획의 상위 규칙이 아니다. 상위 규칙은 Qdrant 1.18 기능의 정점을 찍는 것이다. 에이전트와 LLM은 Qdrant schema 설계, query orchestration, relevance feedback, 평가, 시각화 해석을 강화한다면 제품 런타임에도 포함한다. 단, LLM이 답을 대신 만들어 Qdrant primitive의 역할을 가리면 실패로 본다.

사용 위치 포함 방식 이유 계획 반영
제품 런타임 UI 포함 후보 LLM/챗 UI도 Qdrant 1.18 Query API, feedback, audit, metrics를 조작하고 설명하는 cockpit이면 가치가 있다. “Qdrant Evidence Agent”로 설계한다. 답변보다 query plan, prefetch, formula, rerank, memory impact를 보여준다.
제품 런타임 orchestration 포함 deterministic agent와 LLM-backed agent 모두 가능하다. 핵심은 Qdrant query/migration/eval을 실행 가능하게 만드는가다. ranker, benchmark runner, migration worker, Qdrant query planner, audit/memory inspector agent를 포함한다.
오프라인 인덱싱/라벨링 포함 세션 제목 정제, error_kind 라벨, eval query 생성, sparse token candidate 추출은 Qdrant 검색 품질을 직접 올린다. LLM 생성물은 payload에 provenance, model, generated_at, schema_version을 남겨 Qdrant filter/audit 대상이 되게 한다.
개발 검증 필수 gate 전문가 에이전트 패널은 설계 결함, 성능 리스크, UX 리스크를 병렬로 잡는 데 유효하다. 구현 전 실제 4-way review gate를 추가: Retrieval, Qdrant Ops, UX, Skeptic.
최종 데모/심사 설명 포함 에이전트가 Qdrant 1.18 기능을 더 잘 드러내면 데모에 포함한다. 영상에는 LLM 답변만 보여주지 않는다. query plan, formula term, vector contribution, matrix graph, replay evidence를 함께 보여준다.

수정된 결론

전문가 에이전트와 LLM은 Qdrant 1.18의 정점을 찍는 데 필요하면 제품 계획에 포함한다. 기준은 “챗이냐 아니냐”가 아니라 “Qdrant의 named vectors, hybrid query, multivector, FormulaQuery, relevance feedback, TurboQuant, observability를 더 강하게 드러내느냐”다.

새 검증 Gate

Phase 0 뒤에 실제 전문가 검증을 추가한다. 각 리뷰는 독립 산출물을 남긴다: Retrieval 품질, Qdrant 1.18 기능 적합성, UX/시각화, 반대자 리스크 목록. 네 리뷰에서 같은 위험이 두 번 이상 나오면 roadmap을 수정한다.

사전 조사 반영

현재 저장소 문서와 Qdrant 공식 문서/릴리즈를 함께 기준으로 잡았다. 특히 Qdrant 1.18은 “고도화 계획”에 바로 넣을 수 있는 schema mutation, quantization, 관측 기능을 추가했다.

조사 항목 확인 내용 Memex 적용 판단
현재 Memex 구조 세션당 한 Qdrant point, 5개 dense named vectors(content/tool/path/error/code), lean payload, Replay는 JSONL 재파싱. 이 구조는 유지한다. 추가 세부 검색은 보조 collection으로 분리해 session payload를 부풀리지 않는다.
Qdrant 1.18 릴리즈 TurboQuant, collection memory monitoring, named vector add/remove, audit query/tracing, per-collection metrics, strict mode guardrail이 추가됨. 마이그레이션 UI, 진단 패널, 운영 safety gate, quantization A/B 실험의 핵심 근거로 사용.
Query API / Hybrid prefetch 기반 다단계 검색, RRF/DBSF fusion, sparse+dense 조합, nested prefetch 가능. 현재 Rust weighted combine을 유지하되, 서버사이드 FormulaQuery path를 추가해 “Qdrant Query Cockpit”을 만든다.
Multivector / Late Interaction ColBERT류 token vector를 multivector field로 저장하고 HNSW m=0으로 rerank 전용 운용 가능. Top-K 후보를 ColBERT multivector로 재채점해 “왜 이 세션인가”를 token heatline으로 보여준다.
Search Relevance FormulaQuery, decay, MMR, relevance feedback query가 제공됨. Relevance feedback은 1.17부터 제공. 사용자 클릭/Replay 체류/pos-neg와 LLM/에이전트 라벨러를 모두 oracle 후보로 사용한다. 핵심은 feedback이 Qdrant query를 실제로 개선하는지다.
Payload/Indexing keyword, integer, float, bool, geo, datetime, uuid, text 타입과 on-disk/tenant/principal index가 있음. project/session/time/error/tool path 필드를 index-grade로 정리하고, 자주 쓰지 않는 대형 text index는 on-disk로 둔다.
Topology primitives search_matrix_pairs는 sampled pairwise neighbor graph를 반환한다. 기존 MST를 유지하면서 vector별 multi-layer graph, bridge/gap insight, time scrub을 추가한다.

목표 아키텍처

핵심은 memex_sessions를 안정적인 상위 인덱스로 두고, 필요할 때만 고해상도 보조 collection을 붙이는 계층형 설계다.

Layer 0 · Source ~/.Codex/projects/**/*.jsonl 또는 현 parser 경로. 원문은 Qdrant에 과적재하지 않는다.
Layer 1 · Session Index 기존 memex_sessions. 세션 point id는 uuid_v5 유지.
Layer 2 · Detail Index 선택적 memex_turns, memex_actions. payload는 source pointer, turn index, compact tags만.
Layer 3 · Query Cockpit Query API prefetch, fusion, FormulaQuery, MMR, Discover, relevance feedback orchestration.
Layer 4 · Visual Surfaces Topology galaxy, Replay x-ray, Lens Lab, Prediction rail, Memory thermal panel.

권장 collection schema

memex_sessions
  point_id: uuid_v5(session_id)
  dense vectors:
    content, tool, path, error, code                 # current invariant
    intent_v2, diff_v2, decision_v2                  # add in-place on Qdrant 1.18
  sparse vectors:
    lexical_sparse, path_sparse, command_sparse      # BM25/SPLADE/miniCOIL track
  multivectors:
    colbert_content_tokens                           # HNSW m=0 rerank-only
  payload indexes:
    project_name keyword, git_branch keyword
    ai_title text, start_ts integer, start_iso datetime
    has_errors bool, tool_names keyword[], error_kind keyword
    schema_version keyword, vector_ready keyword[]

memex_turns optional
  point_id: uuid_v5(session_id + ":" + turn_index)
  vectors: turn_content, turn_code, turn_error
  payload: session_id, source_path, turn_index, role, tool_names

memex_actions optional
  point_id: uuid_v5(session_id + ":" + tool_call_id)
  vectors: command_sparse, action_dense
  payload: session_id, turn_index, tool_name, file_ext, exit_error

기술적 킥

이 계획의 핵심은 “Qdrant를 썼다”가 아니라 “Qdrant의 어려운 기능이 하나의 사용 경험 안에서 검증 가능하게 드러난다”는 점이다.

Query APIFormula

Weighted Lens 2.0

현재 Rust combiner는 explainability가 좋다. 여기에 Query API prefetch 5개와 $score[0] 기반 FormulaQuery를 추가해 서버사이드 weighted score를 제공한다. UI는 Rust와 Qdrant formula 결과를 나란히 비교하는 “rank diff” 패널을 가진다.

SparseHybrid

Dense + Sparse + Path Exactness

파일명, CLI flag, crate/module 이름은 dense embedding만으로 약하다. path_sparse, command_sparse, text payload index를 추가해 정확 토큰과 의미 검색을 RRF/DBSF로 합친다.

MultivectorLate Interaction

ColBERT Rerank Track

top 80 후보는 dense/sparse로 빠르게 뽑고, 최종 12개는 multivector MaxSim으로 재정렬한다. 384d BGE의 넓은 recall과 ColBERT의 token-level precision을 분리한다.

Relevance FeedbackAgent Eval

User-Behavior Feedback Query

Mix pos/neg, Replay jump, card dwell, dismiss를 relevance oracle로 저장한다. 이후 Qdrant relevance feedback query로 다음 검색의 vector traversal 자체를 보정한다. 전문가 에이전트나 LLM 라벨러도 oracle로 포함해 feedback score를 만들 수 있다. 단, 그 산출물은 Qdrant payload/filter/audit으로 추적 가능해야 한다.

MMRDiversity

Anti-Duplicate Session Ranking

비슷한 세션이 한 프로젝트 안에서 몰려 나오는 문제는 MMR로 줄인다. “가장 관련 있는 1개 + 서로 다른 맥락 5개”를 default result pack으로 만든다.

MatrixGraph

Multi-Lens Topology

search_matrix_pairs를 content/tool/error/code별로 실행해 edge layer를 만든다. MST는 backbone으로 유지하고, cross-project bridge와 missing bridge를 별도 layer로 표시한다.

TurboQuantBenchmark

Quantization Lab

Qdrant 1.18 TurboQuant bits4를 기본 후보로 삼고 scalar/binary와 비교한다. BGE-small은 384d라 1-bit binary를 바로 쓰지 않는다. recall@10과 p95 latency가 gate다.

Strict ModeMetrics

Operational Guardrails

strict mode로 unindexed filter, batch size, memory pressure를 제한한다. /metrics?per_collection=true와 memory API를 앱의 “Qdrant health rail”에 노출한다.

Schema MutationZero Rebuild

In-Place Vector Migration

Qdrant 1.18의 named vector add/remove로 content_v2를 추가하고 background fill 후 UI traffic을 전환한다. 완전 collection rebuild는 fallback으로만 둔다.

단계별 실행계획

각 단계는 “기능 추가”가 아니라 검증 가능한 gate를 가진다. 시각적 와우는 마지막에 덧칠하지 않고, retrieval primitive가 동작하는 순간부터 같이 만든다.

Phase 0 · 0.5d

Baseline Freeze

Qdrant 1.18.0, qdrant-client, fastembed-rs 버전을 고정하고 현재 5-vector 검색 결과를 golden set으로 저장한다.

Gate: 20개 대표 query의 top-10, latency, memory baseline 저장.
Phase 0.5 · 0.5d

Expert Agent Review Gate

구현 전에 Retrieval, Qdrant Ops, UX, Skeptic 네 관점의 독립 리뷰를 실행한다. 이 단계는 제품 런타임이 아니라 개발 검증이다.

Gate: 네 리뷰 산출물과 반영/기각 사유를 roadmap 옆에 기록.
Phase 1 · 1.5d

Schema Expansion Without Rebuild

Qdrant 1.18 named vector add API를 사용해 intent_v2, diff_v2, sparse placeholders, schema metadata를 추가한다.

Gate: 기존 point id 유지, re-index idempotent, old UI 무중단 동작.
Phase 2 · 2d

Hybrid Query Cockpit

Query API prefetch + RRF/DBSF + FormulaQuery를 Rust command로 노출하고, 기존 Rust combiner와 결과 차이를 비교한다.

Gate: weighted lens 결과 nDCG가 baseline 이상, p95가 허용치 이내.
Phase 3 · 2.5d

Sparse + Exact Token Layer

path/command/code token에 sparse vector 또는 full-text payload index를 붙이고 path-heavy query를 개선한다.

Gate: 파일명, crate, CLI flag query에서 dense-only 대비 실패 케이스 70% 이상 개선.
Phase 4 · 3d

ColBERT Multivector Rerank

Rust ORT adapter 또는 지원 가능한 FastEmbed path로 token multivector를 생성한다. Qdrant multivector HNSW는 m=0으로 저장한다.

Gate: top-80 rerank p95 acceptable, memory budget 통과, token heatline UI 표시.
Phase 5 · 2d

Feedback + MMR + Prediction

pos/neg, Replay jump, dismiss를 local feedback store에 기록하고 Qdrant relevance feedback query와 MMR ranking을 붙인다.

Gate: 중복 결과 감소, dismissed error 재등장률 감소, agent/LLM oracle 포함 시 provenance 검증.
Phase 6 · 2.5d

Topology 2.0 + Replay X-Ray

matrix pairs multi-layer graph, bridge/gap cards, vector contribution timeline, action heatmap을 구현한다.

Gate: desktop/mobile viewport screenshot 검증, graph nonblank, node click/replay roundtrip 정상.
Phase 7 · 1.5d

Ops Hardening

TurboQuant/scalar A/B, memory monitoring, per-collection metrics, audit trace id, strict mode preset, snapshot version check를 연결한다.

Gate: benchmark report 자동 생성, snapshot export/import, strict mode failure UX 확인.
Phase 8 · 1d

Demo Sequence

“search가 아닌 공간 조작” 순서로 3분 데모를 구성한다: topology → mix → replay → recall → lens reactor → memory thermal.

Gate: 영상에 Qdrant primitive 이름과 화면 반응이 동시에 보임.

포함/제약 기준

합의로 강한 아이디어를 버리지 않는다. 아래는 무엇을 버릴지의 목록이 아니라, 어떤 조건으로 포함할지와 어떤 항목이 별도 의사결정을 요구하는지의 기준이다.

별도 결정 필요

  • 제품 기본 런타임의 LLM 호출, chat box, “ask a question” 중심 화면: 포함 가능하다. 단, 범용 QA가 아니라 Qdrant 1.18 query cockpit/evidence agent로 재설계해야 한다.
  • 세션 전체 transcript를 Qdrant payload에 넣어 Replay roundtrip을 줄이는 설계: lean payload 원칙 변경이 필요하다.
  • 검증 없이 1-bit binary quantization을 384d BGE-small에 적용하는 결정: recall/latency gate 통과가 먼저다.
  • Cloud-only inference를 core desktop 기능의 필수 경로로 만드는 결정: local-first 배포 원칙 변경이 필요하다.
  • origin이 ComBba/memex인지 확인하지 않은 git push/PR/gh 작업: 저장소 안전 규칙상 실행하지 않는다.
